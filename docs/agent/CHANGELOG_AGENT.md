# Agent Change Log

## v0.9.84 — 2026-07-17 (유튜브 iframe 포커스 도둑 — 모든 단축키가 죽던 원인)

### CHG-20260717-124 — FIX — iframe에 포커스가 가면 앱 단축키가 전멸
Change: 사용자 보고 — 연습2에서 **Alt+←/→가 크롬 뒤로/앞으로를 실행**하고
**Alt+Shift+←/→는 아무 반응이 없었다**. ADR-0014가 NOT VERIFIED로 남겨 둔
"preventDefault가 크롬을 이기나" 항목이 터진 것처럼 보였지만 **원인이 달랐다.**

**결정적 단서는 Alt+Shift+←/→였다.** 그건 크롬 단축키가 아니다 — 그런데도
아무 일도 안 일어났다면 "preventDefault가 졌다"가 아니라 **우리 핸들러가 아예
안 돌고 있다**는 뜻이다. 브라우저 대조 실험으로 확정:

| 포커스 위치 | keydown 도달 |
|---|---|
| BODY | ✅ 1건 (`alt=true, ArrowRight`) |
| **IFRAME** | ❌ **0건** |

유튜브 영상을 한 번이라도 클릭하면 포커스가 iframe(교차 출처)으로 넘어가고,
그때부터 이 문서의 keydown 리스너에는 **한 건도 오지 않는다**. 즉 Alt만이 아니라
**Tab·Enter·Alt+Z까지 모든 단축키가 함께 죽어 있었다.** Alt+화살표만 유독 티가 난
이유는 그것만 크롬이 가로채 "작업 중 페이지 이탈"이라는 눈에 보이는 사고를 내기
때문 — **원래부터 있던 더 큰 버그를 Alt 전환이 드러낸 것**이지 Alt 선택의 잘못이
아니다. (그래서 **재렌더도 불필요** — "알트" 나레이션은 그대로 유효하다.)

수정: 포커스가 iframe으로 가면 즉시 되찾는다(`window` blur 리스너 + 1초 interval
백스톱). 임베드가 `disablekb: 1`이라 **iframe 안의 포커스는 순손실**이다 —
유튜브 자체 키보드도 꺼져 있어 거기 포커스가 있어야 할 이유가 하나도 없다.
`blur()`만 쓰고 `window.focus()`는 쓰지 않아, 사용자가 다른 앱에 가 있어도 창을
끌어오지 않는다(돌아오면 오히려 단축키가 살아 있는 상태).
Validation: 같은 실험 재현 — 수정 전 iframe 포커스에서 도달 **0건**, 수정 후
`IFRAME → BODY` 되찾기 확인 + 같은 키 **1건 도달**. 빌드 클린.
작업 실수 기록: 처음엔 "다른 앱으로 갔을 때 창을 뺏지 않으려고" `document.
hasFocus()` 가드를 넣었는데, **자동화 창이 OS 포커스를 안 가져 항상 false → 되찾기
자체가 영영 안 도는** 상태였다(실측으로 발견). `blur()`만 하는 이상 그 가드가
지킬 것이 없어 제거. 방어 코드를 넣을 땐 그게 정상 경로를 막지 않는지부터 볼 것.
남은 확인(사용자): 실제 크롬에서 영상 클릭 후 Alt+←/→가 이탈 없이 3초 이동하는지.

## v0.9.80 — 2026-07-17 (조합키 분리: Alt = 이동, Ctrl = 글자 — ADR-0014)

### CHG-20260717-120 — UX — 탐색을 Ctrl → Alt로 (낱말 점프 되찾기)
Change: v0.9.79는 편집칸 안 탐색을 얻는 대가로 **낱말 점프를 버렸다**. 사용자
반박: "나중에 절대 단어 점프도 필요할텐데.. Ctrl을 alt로 바꾸면 안되나?" 맞다 —
탐색을 Alt로 옮기면 **둘 다** 가진다. 채택(ADR-0014).

부수 효과가 본전보다 크다: 종전 나레이션은 "컨트롤과 화살표는 영상, 알트와
화살표는 자막"이라는 **서로 모순된 두 규칙**을 가르쳤다. 이제 **"알트와 화살표는
모두 이동"** 하나다 — 고령 검수자 대상이라 이게 실질적 이득. `Alt+Z`(작업 되돌리기)
↔ `Ctrl+Z`(글자 되돌리기)는 이미 이 규칙과 일치했다(규칙이 없었을 뿐).

| Alt = 이동 | Ctrl = 글자 |
|---|---|
| `←/→` 영상 3초 · `Shift+←/→` 10초 | `←/→` **낱말 점프(복원)** |
| `↑/↓` 자막 · `Shift+↑/↓` 미검수 | `Z` 글자 되돌리기 |
| `Z` 작업 되돌리기 | `Enter` 나누기 · `Shift+Enter` 합치기 |

범위: `Editor.tsx`(핸들러·도움말·말풍선 4개) + 대본 `연습2` L5·L7·L9 컨트롤→알트,
**L12에 규칙 문장 추가** + `visuals.ts` 키캡 3개. **연습2 재렌더 필요**(사용자
승인) → 렌더 후 `TUTORIAL_CHECKPOINTS.playback` 갱신 뒤따라야 함. 다른 코스 무관
(`Ctrl+Enter`·`Ctrl+Shift+Enter`는 불변).

(1:17.4→1:20.4), `Alt+←` **−3.0초**(원위치), 이벤트 `key=ArrowRight target=TEXTAREA`
확인. `Ctrl+→`는 **`defaultPrevented=false`로 끝까지 통과** = 우리가 더는 안 가로챔.
빌드 클린, 잔여 `Ctrl+화살표` 문구 0건, verify_harness OK.
NOT VERIFIED (사용자 실기 확인 요청): ① 실제 크롬에서 `preventDefault`가 `Alt+←`
뒤로가기를 막는지 — **내장 브라우저 창이 `Alt+←` 뒤로가기를 아예 구현하지 않아
대조 실험이 성립하지 않는다**(핸들러 없는 페이지에서도 안 넘어감). 종전 코드도

② 낱말 점프 실동작 — CDP가 네이티브 편집 커맨드를 못 만들어 직접 확인 불가(위

작업 실수 기록: 첫 브라우저 테스트에서 하네스에 `alt+Left`로 키를 보냈더니
