# ADR-0011 — 한자어 처리 순서: API 맞춤법이 정규화 → 무-API 한자 채우기가 병기

- Status: Accepted (1층 역할은 ADR-0015가 보강 — 병기 후보 기록 추가)
- Date: 2026-07-17
- Area: correction / editor / learning
- Supersedes: —
- Amended by: [ADR-0015](ADR-0015-hanja-hint-from-spellcheck.md) — 1층(맞춤법)이
  "정규화만"에서 "정규화 + special 병기 후보 기록"으로 확장. 2층에 규칙 C(힌트
  소비) 추가. 3층 구조 자체는 불변.

## Context

한자 병기(`漢 한자 채우기`)는 **무-API 결정적 치환**이다: HanjaTerm 사전에 있는
표기를 정규식 단어 경계로 찾아 `용어(漢字)`로 바꾼다. 비용 0원이라 검수자에게
열려 있다 (맞춤법 검사는 API라 관리자 전용 — CURRENT_STATE 정책).

관측된 한계 (2026-07-17, 사용자 지적 + 확인):

1. **사전에 있어도 못 잡는다.** `무유지등야(無油之燈也)`는 사전에 있고 count=4인데,
   STT가 `"오도무유지 등야"`처럼 띄어쓰기를 흔들어 놓으면 단어 경계가 깨져
   정규식이 매칭에 실패한다. 즉 실패 원인이 "사전에 없음"만이 아니다.
2. **사전이 얇으면 못 잡는다.** 사람이 손으로 `단어(漢字)`를 달아도 그건 그 셀의
   텍스트로 끝이고 사전에 등록되지 않아 다음 영상에서 또 손으로 달아야 한다.
3. 반면 **한 셀에 여러 개 병기**는 이미 정상 동작한다 (`multi_re.sub()`는 모든
   매치를 치환 — 3개 넣은 테스트 통과). 안 되는 것처럼 보이는 셀은 둘째 용어가
   `tier=common`(자동 병기 제외)이거나 사전에 없는 경우다.

## Decision

한자어 처리를 **두 층으로 분리**하고, 실제 검수 순서(맞춤법 → 한자 채우기)에 맞춘다.

### 1층 — API 맞춤법 검사 (관리자, 1차): **정규화**

맞춤법 검사에 "한자어 정규화" 역할을 함께 얹는다. 병기를 여기서 하지 않는다.
- 고유 용어가 STT 때문에 **쪼개지거나 붙거나 순서가 흔들린 것**을 원래 표기로 되돌린다
  (`오도무유지 등야` → `오도무유지등야`). 보호 용어 목록(`domain_terms`, 1026종)이
  이미 프롬프트에 들어가므로 판단 근거가 있다.
- 사투리·구어는 여전히 건드리지 않는다 (규칙 2/2-1, ADR 없음 — 프롬프트 규칙).

### 2층 — 漢 한자 채우기 (검수자, 2차): **병기**

무-API 결정적 치환 그대로. 1층이 표기를 정규화해 준 덕에 **같은 사전으로 더 많이 잡는다**.

### 3층 — 학습 (📚): **사전 강화**

사람이 손으로 단 `단어(漢字)`를 흡수 단계에서 감지해 HanjaTerm에 등록한다.
사전이 두꺼워질수록 2층이 자동으로 더 많이 잡는다 — 우로보로스가 한자에도 도는 것.

## Rationale

- **비용**: 병기를 API로 하면 매 영상 전 줄이 과금 대상이다. 병기는 결정적 작업이라
  API가 필요 없다. API가 필요한 건 "이게 그 용어인가" 하는 **판단**뿐 →
  판단(1층)만 API, 기계적 치환(2층)은 무-API. 검수자 정책(무-API만)도 유지된다.
- **순서**: 사용자의 실제 작업 순서가 이미 맞춤법 → 한자 채우기다. 1층이 앞에 있으니
  2층이 그 결과를 그대로 먹는다. 순서를 바꿀 필요가 없다.
- **자기 강화**: 3층이 사전을 키우면 2층 적중률이 오르고, 2층이 잡을수록 1층이
  할 일이 준다 (API 비용 감소). ADR-0002(DB=원본)와 같은 방향.

## Consequences

- 맞춤법 프롬프트에 "고유 용어 분절 복원" 규칙 추가 → `PROMPT_VERSION` 상향
  (캐시 무효화 = 재검사 시 1회 재과금).
- 1층이 잘못 붙이면 2층이 엉뚱한 병기를 만든다 → 둘 다 **사람 확인 모달**을 거치므로
  자동 반영되지 않는다 (현행 유지). 특히 동음이의(보시는/布施) 함정은 2층 모달에서 거른다.
- 3층 자동 등록은 오탈자 병기까지 학습할 위험 → 등록 조건을 좁힌다
  (한글 2자 이상 + 괄호 안이 한자만 + 검수 완료 셀). `tier`는 기본 `common`으로
  넣고 흑판 특수어 승격은 사람이 (자동 병기 폭주 방지).
- `tier=common`은 자동 병기 대상이 아니다 — 사전에 있어도 안 채워지는 것이 정상.

## Revisit Trigger

- 1층 정규화가 오히려 오교정을 만들면 → 정규화를 무-API 규칙(사전 기반 퍼지 매칭)으로 내린다.
- HanjaTerm이 수천 종으로 커져 alternation 정규식이 느려지면 → 트라이/Aho-Corasick.
- 3층 자동 등록의 오등록률이 높으면 → 승인 큐(`approved=False`)로 돌린다.
