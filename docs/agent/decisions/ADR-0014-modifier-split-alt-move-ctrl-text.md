# ADR-0014 — 조합키 분리: Alt = 이동, Ctrl = 글자

- Status: Accepted
- Date: 2026-07-17
- Area: editor/ux / tutorial
- Decision Owners: User (Alt 전환 제안 + 재렌더 승인) + Agent (충돌 분석·구현·검증)

## Context

탐색(3초/10초 이동)이 `Ctrl+←/→`에 있었고, 이건 **글상자 안 낱말 점프**(브라우저
기본)와 같은 키다. 한 키를 둘이 못 쓰니 그동안 `!isTypingTarget` 가드로 갈랐다 —
글상자 안에서는 탐색을 포기하고 낱말 점프를 살렸다.

그 결과가 사용자 보고로 터졌다: **"앞으로 3초"만 대안이 없었다.** 뒤로는
`Shift+Tab`이 가드 없이 동작해서 타이핑 중에도 됐지만, 앞으로는 마우스를 잡고
빈 곳을 클릭한 뒤에야 누를 수 있었다. 자막 검수는 듣고-고치고-넘기고를 초 단위로
반복하는 일이라 이 한 번의 마우스 왕복이 리듬을 끊는다.

v0.9.79에서 가드를 제거해 **낱말 점프를 버리고 탐색을 택했다.** 사용자가 바로
반박했다: **"나중에 절대 단어 점프도 필요할텐데."** 맞다 — 긴 자막을 고칠 때
낱말 단위 이동은 결국 필요하고, 그때 가서 되돌리면 이미 배운 키를 또 바꿔야 한다.

사용자 제안: **"Ctrl을 Alt로 바꾸면 안되나?"**

## Decision

**탐색을 Alt로 옮긴다.** 조합키마다 뜻을 하나씩 준다:

| 조합키 | 뜻 | 키 |
|---|---|---|
| **Alt** | **이동** | `←/→` 영상 3초 · `Shift+←/→` 영상 10초 · `↑/↓` 자막 · `Shift+↑/↓` 미검수 자막 · `Z` 작업 되돌리기 |
| **Ctrl** | **글자** | `←/→` 낱말 점프(네이티브 복원) · `Z` 글자 되돌리기 · `Enter` 나누기 · `Shift+Enter` 합치기 |

둘 다 얻는다. 탐색은 **글상자 안에서도** 되고, 낱말 점프도 살아난다.

부수 효과가 오히려 본전보다 크다 — **나레이션이 규칙 하나로 줄어든다.** 종전에는
"컨트롤과 화살표는 영상, 알트와 화살표는 자막"이라는 **서로 모순된 두 규칙**을
가르쳤다. 이제 **"알트와 화살표는 모두 이동"** 하나다. 고령 검수자 대상이라 이
단순화가 이 결정의 실질적 이유다. `Alt+Z`(작업 되돌리기) ↔ `Ctrl+Z`(글자
되돌리기)는 이미 이 규칙과 일치했다 — 규칙이 없었을 뿐이다.

예외로 남기는 것: `Ctrl+\`(이 자막 처음부터 재생)는 재생인데 Ctrl에 있다.
`Alt+\`가 이미 발화 맞춤이라 옮길 자리가 없다. 규칙의 흠으로 인지하고 둔다.

## Consequences

- **연습2 재렌더 필요** (사용자 승인). 대본 3줄(L5·L7·L9) 컨트롤→알트, L12에 규칙
  한 문장 추가, `visuals.ts` 키캡 3개. 렌더하면 `timing.json`이 바뀌므로
  `TUTORIAL_CHECKPOINTS.playback` 갱신이 뒤따라야 한다. 다른 코스는 무관 —
  `Ctrl+Enter`(나누기)·`Ctrl+Shift+Enter`(합치기)는 그대로다.
- **`Alt+←/→`는 크롬·파이어폭스의 뒤로/앞으로 가기 단축키다.** `preventDefault`로
  막는다. 이건 새 가정이 아니다 — 종전 코드도 stray press 방어로 `Alt+←/→`를
  이미 `preventDefault`로 삼키고 있었고, 그 가정이 틀렸다면 종전에도 이미 이탈이
  일어나고 있었다는 뜻이다.
- **파일럿에서 터졌고, 원인은 이 결정이 아니었다** (2026-07-17, v0.9.84로 수정).
  사용자 실기: `Alt+←/→`가 크롬 뒤로/앞으로를 실행, `Alt+Shift+←/→`는 무반응.
  후자가 결정적 단서였다 — 크롬 단축키가 아닌데도 무반응이면 "preventDefault가
  졌다"가 아니라 **핸들러가 아예 안 돈다**는 뜻. 실측 결과 **유튜브 iframe에
  포커스가 있으면 이 문서의 keydown에 이벤트가 0건 도달**한다(BODY 1건 / IFRAME
  0건). 즉 영상을 한 번 클릭하면 Tab·Enter·Alt+Z까지 **모든** 단축키가 죽어
  있었고, Alt+화살표만 크롬이 가로채 눈에 보이는 사고를 냈을 뿐이다. **이 ADR의
  키 선택 문제가 아니라 그 이전부터 있던 포커스 버그**이므로 Alt 유지 + 재렌더
  불필요. 수정 = iframe 포커스 즉시 회수(CHG-20260717-124).
- **VERIFIED (2026-07-17, 사용자 실기)**: 포커스 회수(v0.9.84) 후 **영상을 클릭한
  뒤에도 `Alt+←/→`가 이탈 없이 3초씩 이동**한다. 이로써 이 ADR이 세운 두 전제가
  모두 실기로 확인됐다 — ① 포커스가 이 문서에 있으면 이벤트가 우리에게 오고,
  ② **`preventDefault`가 크롬의 `Alt+←/→` 뒤로/앞으로를 실제로 이긴다**. ②는 이
  개발 환경에서 끝내 검증할 수 없었고(내장 브라우저가 `Alt+←` 뒤로가기를 구현
  하지 않아 대조가 성립 안 함) 사용자 실기가 유일한 답이었다. Alt 채택은 유지.
- **NOT VERIFIED (남음)**: 편집칸 안 `Ctrl+←/→` 낱말 점프 실동작 — CDP가 네이티브
  편집 커맨드를 못 만들어 직접 확인 못 했다. 이벤트가 `defaultPrevented=false`로
  끝까지 통과하는 것은 확인했으므로 브라우저 기본 동작이 실행된다는 뜻이다.
- Mac에서는 `Option+←/→`가 낱말 점프라 그쪽 관례를 뺏는다. 대상 사용자층이
  윈도우라 감수한다 (Mac 뒤로가기는 `Cmd+[`라 이탈 위험은 없다).
- 한글 키보드의 **오른쪽 Alt는 한자 키**로 매핑되는 배열이 있어 `altKey`가 안 뜰
  수 있다. 왼쪽 Alt는 정상. 기존 `Alt+Z`·`Alt+H` 등이 이미 같은 조건이라 새로
  생기는 위험은 아니다.

## Revisit

- 실기에서 `Alt+←`가 브라우저 뒤로가기를 유발하면 즉시 재검토 (그 경우 탐색을
  `Alt+,`/`Alt+.` 같은 비화살표로 옮기고 화살표는 자막 이동에만 남긴다).
- 오른쪽 Alt 미동작 신고가 반복되면 왼쪽/오른쪽 Alt 구분 안내 또는 대체 키.
- `Ctrl+\`을 규칙에 편입할 자리가 생기면 (예: 발화 맞춤을 다른 키로) 정리.
