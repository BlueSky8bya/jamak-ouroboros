# ADR-0016 — 漢 채우기 관리자 전용 + 문맥 발굴 후보 (검증 필수)

- Status: Accepted (Phase 1 구현, Phase 2 계획)
- Date: 2026-07-17
- Area: correction / editor / learning / security
- Decision Owners: User (방향·범위·검증 필수 확정) + Agent (설계·구현)
- Extends: [ADR-0015](ADR-0015-hanja-hint-from-spellcheck.md) (병기 힌트),
  [ADR-0011](ADR-0011-hanja-pipeline-order.md) (한자 3층)

## Context

ADR-0015로 맞춤법이 **사전에 있는 special 낱말**의 병기 후보를 기록한다. 사용자는
이걸 두 방향으로 더 밀고 싶어 한다:

1. **사전에 없어도** — 맞춤법 API가 "이 강연에서 흑판에 쓰인(또는 한자 병기가
   유익한) 한자어인가"를 문맥으로 판단해 후보를 발굴.
2. 후보가 여러 개면(동음이의) **사람이 검증하거나 후보 중 선택**.

핵심 리스크: **한자는 정답이 있는 영역이다.** API가 사전 없이 한자를 *생성*하면
환각한다("정"→精/正/定/靜…). 지금은 사전이 정답 앵커라 환각이 차단된다. 발굴은
앵커를 약화하므로 **자동 병기는 금물** — 사람 검증이 안전장치다(사용자 동의).

## Decision

### 관리자 전용 (Phase 1 — 이 ADR로 구현)

**漢 채우기 전체를 관리자 전용으로 격상한다.** 병기는 정답 판단이라 문맥 발굴·후보
선택이 검수자(어르신) 몫이 아니다(사용자 결정 "2번=관리자만"). 무-API여도 관리자
도구다.
- 서버: `/fill-hanja`에 `_require_admin` (403).
- 프론트: 漢 채우기 버튼을 `isAdmin`일 때만 렌더(맞춤법 버튼과 동일 패턴).
- 검수자는 **관리자가 정리해 둔 병기를 결과로 볼 뿐** 직접 채우지 않는다.

### 발굴 후보 + 검증 (Phase 2 — 계획, ACTIVE_PLAN)

맞춤법이 **두 종류**를 기록한다:
- **(a) 사전 매칭** — special에 있음. 지금(ADR-0015)처럼 결정적 자동 채움.
- **(b) 발굴 후보** — 사전 밖. "흑판 강조어에 준하는 한자어"라고 API가 판단 +
  한자 후보 1~N개. **자동 병기 안 함.** 漢 채우기 UI에서 관리자가 **[이 한자
  맞음 / 후보 중 선택 / 무시]**.

승인 순환:
```
맞춤법 (b) 발굴 → 漢 채우기 UI 후보 → 관리자 검증/선택 → HanjaTerm 등록(3층)
                                                       → 다음부터 (a)로 승격
```

### 범위 (사용자 확정)

- **후보 선택 = 관리자만.** 검수자는 안 함.
- **발굴 후보는 검증 필수.** 사람 승인 없이 자동 병기 금지(환각 방지).
- **과잉 방지** — 프롬프트로 "흑판 강조어에 준하는 것만" 강제 + 사람이 최종 거름.

## Consequences

- **환각 차단** — 발굴은 사람 검증을 통과해야 병기·등록된다.
- **사전 자동 성장** — 승인분이 HanjaTerm에 쌓여 우로보로스 루프가 병기 사전을
  키운다. ADR-0015가 지적한 "사전 청소 부담"도 완화된다(발굴 경로가 있으니
  사전이 완벽할 필요 없음).
- **연습6 파급** — 검수자 튜토리얼(연습6)이 漢 채우기를 가르치는 스텝 2개
  (#19 hanja-fill, hanja-apply)와 대본 L17~L19가 **무의미해진다**(검수자에겐 버튼
  없음). **다음 재렌더 때 연습6에서 漢 스텝·대사 제거** 필요. 그전까지는 연습
  영상에서 검수자가 배우지만 실화면엔 버튼이 없는 불일치가 남는다(기록).
- 관리자 세션이 아니면 漢 채우기 403 — 로컬 단일 사용자(auth off)는 admin으로
  통과하므로 개발엔 영향 없다.

## Non-goals / 주의

- Phase 2는 UI 개편(후보 선택 모달), 스키마(후보 N개 + 승인 상태), 프롬프트(발굴
  규칙), 학습 등록 연결을 포함하는 L3다. 별도 ACTIVE_PLAN으로 단계 진행.
- 발굴이 과병기(일상어까지 후보)를 내면 규칙 강화 + 관리자가 거름. 그래도
  자동 병기는 절대 없다.

## Revisit

- Phase 2 착수 시 이 ADR을 Phase별 Status로 갱신.
- 발굴 후보의 승인율이 낮으면(대부분 무시) 발굴 규칙이 과하다는 신호 → 프롬프트
  조정 또는 발굴 비활성.
- 연습6 재렌더로 漢 스텝 정리되면 파급 항목 해소.
