# ADR-0015 — 한자 병기 힌트: 맞춤법 패스가 문맥 후보를 기록, 漢 채우기가 소비

- Status: Accepted
- Date: 2026-07-17
- Area: correction / editor / learning
- Decision Owners: User (설계 제안·제약·오병기 최소 확정) + Agent (구현)
- Relates to: ADR-0011 (한자 3층) — 1층 정의를 확장(supersede 아님, 보강)

## Context

ADR-0011의 2층(漢 채우기)은 **무-API 결정적 치환**이다. HanjaTerm 사전으로
`단어(漢字)`를 채운다. 단일자는 동음이 많아 `(뜻, 음)` 짝이 자막에 나란히 있을
때만 채운다("얼굴 안 자" → "얼굴 안(顔) 자").

**이 규칙이 못 잡는 것: 홀로 나온 단일자.** 십악중죄를 나열하는 "…살생, 투도,
사음, 망어, 기어, 양설, 악구, **탐, 진, 치**"에서 삼독 貪·嗔·癡는 그냥 낱말이다.
`(뜻, 음)` 짝이 없다. 그렇다고 "탐→貪"을 하드코딩하면 **"탐험"·"탐구"의 탐도**
貪이 된다. **1글자는 문맥 없이는 병기할 수 없다** — 그리고 2층은 문맥을 모른다.

한편 1층(맞춤법 API)은 **이미 셀 전체를 문맥까지 읽는다**. "이 '탐'은 십악 나열
속의 貪이다 / '탐험'의 탐이 아니다"를 판단할 수 있다. 그런데 그 판단을 버리고
오타 교정만 돌려준다.

## Decision (사용자 제안)

**맞춤법 패스가 병기 후보를 곁다리로 기록한다. 실제 병기는 여전히 漢 채우기가
무-API로 한다.** ADR-0011 3층 구조는 유지하고 1층의 역할만 넓힌다:

1. **1층(맞춤법)** — 오타·표기 정규화 **+ 병기 후보 기록**. 프롬프트에 규칙 10과
   "강조 한자어" 목록(gloss 있는 special 단일자만)을 주고, 그 낱말이 이 줄에 그
   뜻으로 쓰였으면 `hanja: [{idx, reading, hanja}]`로 남긴다. **텍스트(fixes)는
   절대 바꾸지 않는다** — 병기는 기록만. 같은 소리 다른 뜻이면(탐험의 탐) 기록
   안 함. API 추가 호출 0 (맞춤법 응답에 실림).
2. **저장** — 셀별 `Segment.hanja_hints`(JSON). 맞춤법 캐시(LlmCache)에도
   `hanja` 열로 실어, 캐시 히트 시 힌트가 복원된다.
3. **2층(漢 채우기)** — 규칙 C 추가: 이 셀의 `hanja_hints`를 규칙 A·B와 함께
   적용한다. 낱말이 **아직 셀에 있고 아직 병기 안 됐을 때만**(뒤에 `(` 없음).
   조사 흡수(규칙 B의 JOSA)로 "탐을·치입니다"도 잡고, "탐욕" 내부는 안 건드린다.
   셀당 첫 등장 1회. 여전히 **무-API**.

### 범위 결정 (사용자)

- **special tier만.** 병기 후보는 gloss 있는 **special 단일자**(흑판 강조어)로
  한정한다. 다자어는 2층이 단어 경계로 이미 잘 잡으므로 제외. 일반 한자어까지
  힌트를 내면 병기는 풍부하지만 오병기·비용이 는다 → **오병기 최소**를 택했다.
- 힌트 무효화는 **위치가 아니라 낱말 존재**로 한다 — 기록 후 사람이 텍스트를
  고쳐 그 낱말이 사라지면 규칙 C가 매칭 실패로 조용히 스킵한다.

## Consequences

- 貪·嗔·癡처럼 사전만으론 못 채우던 단독 강조어를, 맞춤법을 한 번 돌린 뒤엔
  漢 채우기 한 번으로 **문맥을 실제로 판단한 것처럼** 병기한다. API는 맞춤법 1회뿐.
- 사람은 그대로 제안-확인(dry) 흐름으로 검토·선택 적용한다 — 결정적·되돌리기 가능.
- 병기 후보 목록(special 단일자)이 바뀌면 맞춤법 system_prompt가 바뀌고, 그게
  캐시 salt라 **자동 재검사**된다 — 새로 등록한 강조어가 다음 맞춤법에 반영된다.
- DB 컬럼 2개 추가(`Segment.hanja_hints`, `LlmCache.hanja`) — `_ensure_columns`
  가산 마이그레이션. 기존 데이터 무해(빈 문자열 기본).

## Non-goals / 주의

- **맞춤법이 병기를 직접 넣지 않는다.** 힌트는 기록일 뿐, 텍스트 변경은 오직 漢
  채우기(사람 확인)에서. 형의 제약: "실제 맞춤법 검사중에 변경하지 말 것(한자어
  병기 관련 부분만)".
- 힌트는 **special 단일자만**. 다자어·common은 종전 규칙(A·B) 그대로.
- 1글자 오병기가 관측되면 프롬프트 규칙 10을 조이거나 해당 항목을 common으로
  강등한다 (병기 후보에서 빠짐).

## Revisit

- 힌트가 과병기(문맥 오판)를 내면: 규칙 10 강화 · 항목 tier 강등 · 힌트에 gloss
  검증 추가.
- 다자어에도 문맥 병기가 필요해지면(동형이의 다자어) 범위를 넓힐지 재검토.
